# Введение

В последние годы получили распространение системы, в которых вычислительное оборудование используется не только для добычи криптовалюты, но и как элемент бытовой или локальной инженерной инфраструктуры. Одним из таких сценариев является применение ASIC-майнеров в качестве управляемого источника тепла. В подобных условиях особенно важны непрерывный мониторинг состояния оборудования, удалённое управление режимами работы, сбор телеметрии, а также расчёт экономической эффективности эксплуатации.

Проект CryptoTerm представляет собой программно-аппаратный комплекс для мониторинга и управления ASIC-майнерами, подключёнными к Raspberry Pi. Система предназначена для наблюдения за температурой, стабильностью работы и историей метрик, а также для управления майнерами через приложение.

# Основные модули

- iOS-приложение CryptoTerm-main;
- backend cryptoterm_backend-refactor;
- web-frontend cryptoterm_frontend-dev;
- edge-часть для Raspberry Pi
- single_home_rp_control-stage-2;
- отдельный модуль rp_restructured_mql-main;
- landing-страница и сопутствующая инфраструктура.

# 1. Назначение проекта

Основное назначение CryptoTerm заключается в создании единой системы, которая позволяет:

- регистрировать и обслуживать устройства на базе Raspberry Pi;
- собирать телеметрию с ASIC-майнеров и внешних датчиков;
- хранить метрики и журналы событий;
- удалённо изменять режимы работы оборудования;
- планировать выполнение управляющих команд;
- рассчитывать ориентировочную доходность майнинга;
- предоставлять пользователю доступ к данным через мобильный и веб-интерфейс.

Иными словами, проект решает сразу две задачи:

техническую — контроль физического оборудования и его состояния;
прикладную — анализ эффективности эксплуатации майнеров как вычислительных и теплогенерирующих устройств.

# 2. Общая архитектура системы

### 2.1 Пользовательский уровень

Итерфейсы, через которые пользователь взаимодействует с системой:

- мобильное приложение для iOS;
- web-панель администратора и мониторинга;
- landing-страница проекта.

### 2.2 Серверный уровень

Центральным узлом системы выступает backend на Spring Boot. Он реализует:

- REST API;
- аутентификацию и авторизацию;
- хранение данных;
- агрегацию метрик;
- расчёт прибыли;
- интеграцию с MQTT;
- маршрутизацию команд к устройствам.

### 2.3 Полевой уровень устройств

На стороне Raspberry Pi располагаются локальные службы, обеспечивающие:

- запуск и перезапуск процессов;
- контроль Wi-Fi;
- основной цикл опроса датчиков и оборудования;
- обмен данными с backend;
- индикацию аварийного состояния через LED;
- работу с подключённым Arduino-датчиком температуры.

### 2.4 Инфраструктурный уровень

В проект также входят сервисы развертывания и сопровождения:

- Nginx;
- Certbot;
- Docker Compose;
- TimescaleDB;
- MongoDB;
- Mosquitto MQTT broker;
- backup-сервис.

# 3. Состав проекта и роль отдельных модулей

### 3.1 Backend: cryptoterm_backend-refactor

Данный модуль является ядром всей системы. Реализован как REST API для мониторинга и управления ASIC-майнерами на базе Raspberry Pi и использует Java 21, Spring Boot 3.3.4, TimescaleDB, MongoDB, MQTT, JWT и Docker.

#### 3.1.1 Аутентификация и безопасность

В проекте присутствуют:

- AuthController;
- AuthService;
- JWT-сервис;
- конфигурация Spring Security.

Это означает, что пользовательская работа организована через защищённый API, а доступ к операциям осуществляется по токенам.

#### 3.1.2 Работа с устройствами

Слой устройств включает сервисы и контроллеры для:

- регистрации Raspberry Pi;
- получения списка устройств пользователя;
- получения подробной информации об устройствах;
- обслуживания майнеров и сенсоров.

В терминах предметной области backend хранит сведения как минимум о:

- пользователях;
- устройствах;
- майнерах;
- температурных датчиках;
- группах сенсоров;
- конфигурациях оборудования.

#### 3.1.3 Сбор и хранение метрик

Backend обрабатывает:

- температуру;
- хешрейт;
- энергопотребление;
- прочие кастомные метрики;
- журналы событий.

Для этого выделен отдельный слой monitoring с контроллерами:

- TemperatureController;
- HashRateController;
- PowerConsumptionController;
- OtherMetricController;
- LogController;
- InstanceMetricsController.

В качестве хранилища временных рядов используется TimescaleDB, что является технически оправданным решением: температура, хешрейт и другие телеметрические показатели формируют последовательности измерений во времени, для которых time-series БД подходит лучше обычной реляционной схемы.

#### 3.1.4 Управляющие команды

Backend поддерживает не только чтение метрик, но и активное управление оборудованием. Для этого реализованы:

- MinerControlController;
- AsicDeviceCommandController;
- AsicCommandService;
- AsicCommandTemplateService.

В системе предусмотрены:

- переключение режимов работы майнера;
- перезапуск оборудования;
- шаблоны команд для ASIC;
- выполнение команд по расписанию.

### 3.2 Web-frontend: cryptoterm_frontend-dev

Frontend-модуль представляет собой веб-клиент, собранный на базе React и Vite.

Web-интерфейс реализует:

- кран входа и регистрации;
- защищённые маршруты;
- домашнюю панель мониторинга;
- работу с API backend;
- автоматическое обновление токена;
- построение графиков и визуализацию метрик.

Особое значение имеет файл api.js, в котором реализованы:

- работа с access token и refresh token;
- повтор запроса после 401;
- приведение форматов полей между frontend и backend;
- вызовы метрик, управления и авторизации.

Следовательно, web-frontend выполняет роль административной панели и аналитического интерфейса. Через него пользователь может не только просматривать состояние устройств, но и управлять конфигурацией системы.

### 3.3 Мобильное приложение: CryptoTerm-main

Мобильная часть проекта реализована для iOS.

В приложении присутствуют следующие функциональные блоки:

- Authentication — вход и регистрация пользователя;
- HomeScreen — главная панель мониторинга;
- Coins — связанный раздел отображения данных;
- MiningCalendar — календарь/планирование команд;
- ProfitServices — расчёт прибыли;
- MetricsServices — загрузка графиков и метрик;
- DeviceServices — работа с устройствами и майнерами.

Мобильное приложение покрывает ключевые пользовательские сценарии:

- контроль текущего состояния;
- просмотр истории;
- удалённое управление;
- планирование режимов работы.

### 3.4 Edge-уровень Raspberry Pi: single_home_rp_control-stage-2

- systemd-сервисы автозапуска;
- скрипты настройки;
- основной цикл опроса;
- клиент взаимодействия с backend;
- файловый обмен между процессами;
- обработка аварий;
- реакция на потерю Wi-Fi;
- световая индикация проблем.

Архитектура локального узла специально сделана устойчивой к сбоям: основной цикл и клиент работают в разных процессах и обмениваются данными через файлы, а не через сокеты. Такое решение упрощает восстановление после отказов и снижает риск взаимных блокировок.

#### 3.4.1 systemd-сервисы

В папке services находятся unit-файлы, отвечающие за:

- настройку Wi-Fi;
- запуск control loop;
- установку зависимостей;
- запуск notification service.

Следовательно, Raspberry Pi в составе CryptoTerm рассматривается как автономный edge-узел, который после развёртывания должен самостоятельно поднимать необходимые сервисы при старте системы.

#### 3.4.2 Notification service

В notification_service/main.py реализован Flask-сервис, управляющий светодиодом Raspberry Pi через GPIO. Он поддерживает режимы:

- включено;
- выключено;
- мигание.

Это позволяет использовать светодиод как простой локальный индикатор состояния.

#### 3.4.3 Arduino и температурные датчики

В проекте присутствует arduino_temperature.ino, где реализовано считывание температур с аналоговых входов Arduino по данным терморезисторов. Программа:

- принимает команду по Serial;
- читает значения на входах A0–A5;
- вычисляет температуру;
- возвращает массив измерений.

Это показывает, что система способна работать не только с данными самого ASIC-майнера, но и с внешними температурными датчиками. С инженерной точки зрения это важно, так как температура в таких системах может контролироваться на разных уровнях:

- температура самого майнера;
- температура воздуха;
- температура отдельных зон установки;
- косвенно — эффективность теплового использования устройства.

# 4. Организация хранения данных

Можно выделить два основных контура хранения.

### 4.1 Реляционный контур

В PostgreSQL/TimescaleDB хранятся:

- пользователи;
- устройства;
- майнеры;
- типы метрик;
- сами метрики;
- пользовательские и служебные логи.

Это позволяет обеспечить строгую структуру сущностей и одновременно эффективно работать с временными рядами.

### 4.2 Документный контур

В MongoDB хранятся:

- конфигурации майнеров;
- шаблоны команд ASIC;
- история команд.

Такое разделение БД выглядит архитектурно оправданным: телеметрия и операционные сущности размещаются в одной модели хранения, а конфигурационные и сценарные документы — в другой.

# 5. Основные бизнес-сценарии работы системы

### 5.1 Регистрация пользователя и вход

Пользователь создаёт учётную запись, проходит аутентификацию и получает токены доступа. После этого ему становятся доступны устройства, связанные с его профилем.

### 5.2 Регистрация Raspberry Pi

Устройство проходит отдельную процедуру регистрации через API. Для этого используется специализированный endpoint и API key. После регистрации сервер начинает ассоциировать устройство с конкретным пользователем.

### 5.3 Передача телеметрии

Локальный узел на Raspberry Pi собирает данные с майнера и внешних датчиков, после чего отправляет:

- метрики;
- обытия;
- журналы;
- служебную информацию о состоянии.

Передача реализована через MQTT и/или через backend-совместимые механизмы обмена.

### 5.4 Анализ и визуализация

Backend сохраняет данные и предоставляет API для их получения. Далее web-frontend и iOS-клиент строят:

- графики температуры;
- графики хешрейта;
- показатели мощности;
- сводную статистику по устройствам;
- историю логов.

### 5.5 Управление оборудованием

Пользователь может удалённо:

- изменить режим работы майнера;
- выполнить шаблон управляющей команды;
- запланировать изменение режима;
- перезапустить оборудование.

### 5.6 Расчёт прибыли

Система за заданный период:

- вычисляет фактическое время работы;
- определяет средний хешрейт и среднюю мощность;
- подгружает сетевые параметры Bitcoin;
- оценивает объём добытого BTC;
- рассчитывает денежный эквивалент.

Такой сценарий особенно полезен для оценки целесообразности конкретных режимов работы.

# 6. Надёжность и отказоустойчивость

Одним из сильных качеств проекта является внимание к устойчивости edge-части.

### 6.1 Изоляция процессов

Основной цикл управления и клиентский процесс разведены по отдельным процессам и обмениваются данными через файлы. Это снижает сложность межпроцессного взаимодействия и упрощает перезапуск отдельных компонентов.

### 6.2 systemd-управление

Использование systemd позволяет автоматически поднимать сервисы после перезагрузки устройства и перезапускать их после аварий.

### 6.3 Световая индикация ошибок

Наличие LED-сервиса на Raspberry Pi позволяет быстро диагностировать неполадки непосредственно на устройстве.

### 6.4 Резервное копирование

В docker-compose backend-модуля присутствует отдельный backup-сервис с ежедневным резервным копированием PostgreSQL и MongoDB. Это повышает эксплуатационную надёжность и упрощает восстановление после сбоев.

# 7. Безопасность системы

В системе реализованы следующие меры защиты:

- JWT-аутентификация;
- refresh token механизм;
- хеширование паролей;
- HTTPS через Nginx и Certbot;
- ограничение доступа к защищённым API;
- API key для регистрации устройств;
- CORS-настройки;
- rate limiting.

Для проекта, который управляет физическим оборудованием удалённо, это особенно важно, поскольку компрометация канала управления может привести не только к утечке данных, но и к изменению режимов работы майнеров.

8. Выводы

CryptoTerm является комплексной распределённой системой мониторинга и управления ASIC-майнерами, ориентированной на домашние и малые инфраструктурные сценарии эксплуатации.

Ключевые особенности проекта:

- аличие серверной части с развитой бизнес-логикой;
- использование time-series хранилища для телеметрии;
- поддержка удалённых управляющих команд;
- наличие нативного мобильного клиента;
- наличие web-панели;
- развитая edge-часть для Raspberry Pi;
- поддержка внешних температурных датчиков и локальной индикации;
- расчёт прибыльности майнинга;
- контейнеризированная инфраструктура развертывания.

Таким образом, проект CryptoTerm можно охарактеризовать как полноценную программно-аппаратную платформу для дистанционного контроля, управления и анализа работы ASIC-майнеров.
